### Известный баг с прокси на этой машине (см. также [[project-net-monitor-dropouts]])
Windows системный прокси = `socks=127.0.0.1:10808` (реестр `HKCU\...\Internet Settings`, это локальный Xray-листенер Happ VPN, слушает как SOCKS5). `httpx` при автоопределении (`trust_env=True`, поведение по умолчанию) неверно трактует его как `socks4://` → падает `ValueError`. Полное отключение прокси (`trust_env=False`) вместо этого шлёт трафик напрямую с реального IP — OpenRouter's WAF блокирует прямые запросы («403 Access denied by security policy»). **Правильный фикс** (в `llm.py`, `_get_client()`): `httpx.Client(proxy="socks5://127.0.0.1:10808", trust_env=False)` — явно указываем верную схему прокси. Если снова увидите 403 "Access denied" или необработанный `ValueError` про прокси — проверьте, не поменялся ли порт Happ-прокси в реестре (`Get-ItemProperty 'HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings'`).

Все файлы написаны и закоммичены: `app.py`, `config.py`, `llm.py`, `storage.py`, `static/index.html`, `static/app.html`, `static/app.js`, `static/style.css`.

## Архитектура (зафиксировано)
Один Python-процесс: FastAPI backend + статичный HTML/JS фронт (без Node/сборки). Один общий логин/пароль на всех (cookie-сессия, `itsdangerous`). Хранение — JSON-файлы на диске, без БД. LLM — через **OpenRouter** (не Anthropic напрямую — у пользователя личная подписка Claude, а не оплаченный API-баланс).

## Функционал MVP — реализован полностью
1. **Вход** — единый логин/пароль из `.env`, cookie-сессия.
2. **Чат по правовым вопросам** — `POST /api/chat`, история в `data/chats/`.
3. **Генерация документов** — `POST /api/documents/generate`, список/скачивание.
4. **Анализ договора** — `POST /api/contracts/analyze` (.txt/.docx/.pdf).

**Сознательно НЕ делаем в MVP** (напоминать не чаще раза в день, если тема всплывёт): мониторинг законов/новостей, поиск дел в VDR, compliance GDPR/CCPA, мульти-агентная оркестрация.

## Ручная проверка (2026-07-13)
Прогнал сервер локально (порт занятости 8000/8001 конфликтовали с другими проектами на машине — итоговый тестовый порт 8010):
- Логин/логаут/сессия — работают корректно (303 → cookie, 401 без cookie, logout чистит сессию).
- Статика (`/`, `/app`, `/static/*`) отдаётся, `/app` защищён `require_session`.
- Чат/документы/анализ договора — запросы доходят до OpenRouter и возвращают **понятную ошибку 502** (не падение), потому что `OPENROUTER_API_KEY` в `.env` — ещё плейсхолдер (`sk-or-v1-xxxxxxxxxxxxxxxx`), OpenRouter отвечает `401 User not found`. Это доказывает, что путь запроса и обработка ошибок работают правильно; сами сценарии (реальный ответ LLM) непроверяемы без реального ключа.

### Найденный и исправленный баг
На машине системно установлена переменная окружения прокси `socks4://127.0.0.1:10808` (от VPN-приложения Happ). `httpx`/`openai`-клиент не поддерживает схему `socks4` (только `socks5`) и падал необработанным `ValueError` при первом реальном вызове LLM — это давало клиенту голый `500 Internal Server Error` вместо понятного сообщения, что нарушало явное требование спеки про обработку ошибок. Исправлено в `llm.py`: клиент OpenAI создаётся с `httpx.Client(trust_env=False)`, плюс добавлен catch-all `except Exception` в `call_llm`, чтобы любая будущая непредвиденная ошибка тоже возвращалась как чистый `LLMError`, а не падение сервера.

### Следующие шаги по улучшению — статус на 2026-07-14 вечер

1. **Расследование дублирующих туннелей — сделано, решение оставлено пользователю (не убивал процессы).**
   Находки из `cloudflared.log`/`ngrok.log`:
   - `cloudflared` — это **anonymous quick tunnel** (`trycloudflare.com`, без конфига в `~/.cloudflared`, без именованного туннеля), URL `https://excluding-anticipated-round-integrating.trycloudflare.com`. За время работы **4132 строки ERR**, в основном постоянные `TLS handshake with edge error: EOF` / `i/o timeout` — похоже на конфликт с локальным VPN/SOCKS-прокси (Happ, см. баг с прокси выше) или блокировку egress UDP:7844. Также замечено **два процесса `cloudflared.exe`** запущены одновременно (PID 8704 и 18420) — вероятно, случайный дубль-запуск, а не намеренно.
   - `ngrok` — стабильный поддомен `https://deflector-reversing-stallion.ngrok-free.dev`, работает с 2026-07-13, тоже есть obрывы (`heartbeat timeout`), но переподключается штатно, ошибок на порядок меньше.
   - **Вывод: `ngrok` выглядит как "настоящий"/предпочтительный туннель** (стабильный URL, меньше ошибок, легче делиться постоянной ссылкой). `cloudflared` похож на побочный эксперимент с нестабильным случайным URL и явным конфликтом с локальной сетью/VPN. Рекомендация пользователю: оставить `ngrok`, убить оба `cloudflared.exe`, либо (если `cloudflared` всё же нужен) исправить сетевой конфликт с Happ VPN аналогично фиксу в `llm.py`. Решение и убийство процессов оставлено пользователю, ничего не трогал.

2. **Rate-limiting — сделано, закоммичено.**
   В `app.py` добавлен простой in-memory rate-limiter (`rate_limit()` dependency, скользящее окно `deque` по session-cookie токену, без сторонних зависимостей типа slowapi — не нужны для одного процесса): лимит **20 запросов / 60 секунд** на сессию, при превышении `429`. Подключен к `POST /api/chat` и `POST /api/documents/generate` как доп. `Depends`. Сбрасывается при рестарте процесса — не проблема для hobby-scale. **Не вступит в силу, пока сервер не будет перезапущен** (по условию задачи процесс не трогал).

3. **Cookie flags — сделано, закоммичено.** `set_cookie` в `/login` теперь ставит `secure=True` (было отсутствовало). `httponly=True` уже было. `samesite="lax"` оставлен как есть — это одностраничное приложение без кросс-сайтовых запросов, `lax` уже достаточно безопасен и не ломает обычную навигацию/редирект после логина (`strict` сломал бы `303`-редирект с `/login`). Приложение торчит наружу только через HTTPS-туннели (cloudflared/ngrok оба терминируют TLS), поэтому `Secure` безопасно всегда, даже для локальной разработки через туннель. Вступит в силу после рестарта сервера.
4. Бэкап `data/` (JSON-хранилище без БД, без резервного копирования). — не сделано.
5. Логротация `cloudflared.log` (уже 1.6MB, на момент проверки — 1.6MB и 4132 ERR-строк). — не сделано.
