# Jurist — оркестратор практик права

## Назначение
Расширение существующего MVP (см. `2026-07-12-jurist-mvp-design.md`): перед
каждым обращением к LLM (чат, генерация документа, анализ договора) определяется
релевантная юридическая практика, и в системный промпт подмешивается
специализированный фрагмент под эту практику. Цель — более точные и предметные
ответы без изменения UX (пользователь не видит, какая практика выбрана).

Это не полноценный мульти-агентный фреймворк с памятью/инструментами/цепочками —
сознательно лёгкий подход в 2 LLM-вызова, вписывающийся в текущий стек без новых
зависимостей.

## Стек (без изменений)
FastAPI, статичный HTML/JS, OpenRouter, JSON-хранилище, без БД — см. MVP-спеку.

## Набор практик
1. `civil` — гражданское/договорное право
2. `labor` — трудовое право
3. `corporate` — корпоративное право (ООО, АО)
4. `family` — семейное/наследственное право
5. `it` — ИТ-право (перс. данные, лицензии ПО, ИТ-услуги)
6. `tax` — налоговое право
7. `admin` — административное право (штрафы, госорганы)
8. `realestate` — недвижимость/земельное право
9. `general` — общая практика (fallback)

## Архитектура

### Новый файл `practices.py`
- Словарь `PRACTICES: dict[str, dict]` — на каждую практику: `id`, `label`
  (для внутренних логов), `system_prompt_fragment` (специализация, типичные
  кодексы/акты, стиль ответа).
- Константа `PRACTICE_IDS = list(PRACTICES)` — используется в промпте классификатора.

### Изменения в `llm.py`
- Новая функция `classify_practice(text: str) -> str`:
  - Отдельный лёгкий вызов OpenRouter (модель — Claude Haiku, как текущий надёжный
    fallback), короткий промпт: "определи практику права из списка {PRACTICE_IDS}
    для следующего текста, ответь только id".
  - Парсинг ответа: если результат не входит в `PRACTICE_IDS` — возвращает `general`.
  - Обёрнута в try/except: любая ошибка (сеть, таймаут, невалидный ответ) →
    возвращает `general`, не пробрасывает исключение выше и не блокирует основной
    запрос (см. "Обработка ошибок" в MVP-спеке — тот же принцип: понятная деградация,
    а не падение).
- Существующая `call_llm` (или её вызовы в `app.py`) получает доп. параметр
  `practice_id`, использует `PRACTICES[practice_id]["system_prompt_fragment"]`,
  добавленный к базовому системному промпту.

### Изменения в `app.py`
Во всех трёх маршрутах (`/api/chat`, `/api/documents/generate`,
`/api/contracts/analyze`) перед основным вызовом LLM добавляется шаг:
1. Взять релевантный текст запроса (текст сообщения / вводные документа /
   текст договора — что уже есть в теле запроса).
2. `practice_id = classify_practice(text)`.
3. Передать `practice_id` в основной вызов `call_llm`.

Классификация не логируется пользователю и не возвращается в API-ответе —
чисто внутренний routing (по решению из брейнсторминга).

## Обработка ошибок
Классификатор никогда не должен приводить к отказу основного сценария:
любая его ошибка → тихий fallback на `general`. Ошибки основного вызова LLM
обрабатываются как и раньше (см. MVP-спеку).

## Тестирование
Без юнит-тестов (та же логика, что в MVP — одноразовое приложение для одного
пользователя). Ручная проверка через `/verify`: для каждой из 3 функций — по
одному запросу, явно относящемуся к разной практике (например, трудовой спор,
корпоративный вопрос, ИТ-договор), сравнить качество ответа до/после.

## Границы (сознательно не делаем)
- Параллельные агенты / multi-agent фреймворки (LangGraph, CrewAI и т.п.)
- Память между практиками, передача контекста от одного агента другому
- Отображение выбранной практики в UI
- Более 9 практик — можно добавить отдельным циклом при появлении запроса

Эти пункты не исключаются навсегда — та же оговорка, что в MVP-спеке.
