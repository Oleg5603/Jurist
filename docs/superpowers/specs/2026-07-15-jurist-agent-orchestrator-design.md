# Jurist — полноценный оркестратор агентов

## Статус относительно прежних решений
Спека `2026-07-13-jurist-practice-orchestrator-design.md` сознательно отказалась
от «полноценного мульти-агентного фреймворка» в пользу лёгкого 2-вызовного
роутинга по практикам. Этот документ **пересматривает то решение** по прямой
просьбе пользователя от 2026-07-15: нужен настоящий оркестратор агентов с
чёткой структурой, связями между агентами и проверкой корректности на каждом
шаге. Роутинг по 9 практикам (`practices.py`) остаётся без изменений — он
становится одним из агентов внутри новой структуры, а не отдельным лёгким
хаком.

Одновременно пользователь подтвердил (2026-07-15, брейнсторминг):
- Все 3 текущие функции (чат, документы, анализ договора) полезны на практике — углубляем их, не меняем.
- Список 9 практик права полный — новые не добавляем.
- Нужны: верификация цитат норм права (галлюцинации номеров статей) и библиотека шаблонов документов.
- Верификация цитат — через свой offline MCP-сервер с индексом статей (не live-поиск в интернете), чтобы не добавлять ещё одну сетевую зависимость поверх уже нестабильной сети пользователя.

## Цель оркестратора
Единая точка входа для каждого из 3 сценариев (чат/документ/договор), которая
явно описывает: какие агенты участвуют, в каком порядке, что каждый агент
гарантирует на выходе, и как проверяется корректность до того, как результат
уйдёт пользователю.

## Структура агентов

```
                         ┌─────────────────────┐
  запрос пользователя →  │   JuristOrchestrator │ →  ответ пользователю
                         └──────────┬───────────┘
                                    │
            ┌───────────────┬──────┴───────┬────────────────────┐
            ▼               ▼              ▼                    ▼
  PracticeClassifierAgent  SpecialistAgent  TemplateSelectorAgent CitationVerifierAgent
  (кл. практики права)     (Chat|Document|   (только для Document) (пост-проверка
                            Contract)                                 цитат норм)
```

### 1. PracticeClassifierAgent (существует, `llm.py:classify_practice`)
- **Вход:** текст запроса (вопрос/вводные документа/текст договора).
- **Выход:** `practice_id` из 9 практик или `general`.
- **Гарантия корректности:** при любой ошибке (сеть, невалидный ответ) —
  тихий fallback на `general`, никогда не бросает исключение наружу.
- **Связь:** результат передаётся в SpecialistAgent как часть системного промпта.

### 2. TemplateSelectorAgent (новый, только для документов)
- **Вход:** `doc_type` (NDA/Договор/Претензия/Исковое заявление).
- **Выход:** текст эталонного шаблона из `templates/<doc_type>.txt` (few-shot
  пример реальной структуры документа) или `None`, если для типа нет шаблона.
- **Гарантия корректности:** чисто файловый lookup, без LLM — не может
  «галлюцинировать». Отсутствие файла — не ошибка, просто `None`.
- **Связь:** шаблон подмешивается в системный промпт SpecialistAgent
  (`DOCUMENT_SYSTEM_PROMPT` + практика + шаблон-пример), стабилизируя
  структуру генерируемого документа.

### 3. SpecialistAgent (существует, один класс на 3 сценария: Chat/Document/Contract)
- **Вход:** практика (от агента 1), опционально шаблон (от агента 2), текст
  запроса, история (для чата).
- **Выход:** черновой текст ответа/документа/анализа.
- **Гарантия корректности:** это единственный агент, которому разрешено
  ошибаться содержательно (LLM) — все его сетевые/API-ошибки превращаются в
  понятный `LLMError` → `502` с русским текстом (см. MVP-спеку). Логическая
  корректность (не выдумал ли он несуществующую статью) — не его
  ответственность, за это отвечает следующий агент.

### 4. CitationVerifierAgent (новый)
- **Вход:** черновой текст от SpecialistAgent.
- **Выход:** тот же текст + список предупреждений вида
  `⚠️ Не удалось подтвердить: ст. 1234 ГК РФ (возможно, ошибка модели)`,
  добавленных отдельным блоком в конец ответа. Текст ответа не переписывается
  и не обрезается — только помечается.
- **Механизм:**
  1. Regex извлекает упоминания статей вида `ст\.?\s*(\d+)\s+(ГК|ТК|НК|КоАП|СК|ЗК)\s*РФ`
     и `Федеральный закон №\s*(\d+)-ФЗ` (плюс варианты полного написания слова «статья»).
  2. Для каждой найденной пары (закон, номер) — вызов инструмента
     `lookup_article` на локальном MCP-сервере `jurist-law-lookup`.
  3. Если `exists: false` или сервер недоступен для конкретной цитаты —
     добавляем предупреждение. Полная недоступность MCP-сервера **не блокирует**
     ответ — при ошибке подключения агент логирует и возвращает исходный текст
     без пометок (fail-open, деградация, а не отказ).
- **Гарантия корректности:** сам не генерирует текст, не может добавить
  ложную информацию — только помечает потенциально ложную.

### 5. JuristOrchestrator (новый, `orchestrator.py`)
Единственная точка, которая знает про порядок вызовов. Три публичных метода,
по одному на существующий API-эндпоинт:

```python
class JuristOrchestrator:
    def handle_chat(self, message: str, history: list[dict], model: str) -> str: ...
    def handle_document(self, doc_type: str, prompt: str, model: str) -> str: ...
    def handle_contract(self, contract_text: str, model: str) -> str: ...
```

Внутри каждого метода — фиксированная последовательность:
`classify → (select_template, если Document) → specialist_call → verify_citations → return`.

`app.py` перестаёт напрямую дёргать `classify_practice`/`call_llm` — вызывает
только `orchestrator.handle_*`, что делает маршруты в `app.py` тоньше и
переносит всю логику связей между агентами в один файл, который можно
тестировать независимо от FastAPI.

## Новый компонент: MCP-сервер `jurist-law-lookup`

### Назначение
Offline-проверка существования статьи закона по номеру — без похода в
интернет (см. обоснование варианта A в брейнсторминге: не добавлять ещё одну
сетевую зависимость поверх уже нестабильной сети).

### Данные
`mcp_servers/law_lookup/law_index.json` — статический индекс, собранный один
раз из открытого источника (например, структуры разделов на pravo.gov.ru или
базового датасета кодексов), формат:
```json
{
  "ГК РФ": {"15": "Возмещение убытков", "395": "Ответственность за неисполнение денежного обязательства", ...},
  "ТК РФ": {"81": "Расторжение трудового договора по инициативе работодателя", ...},
  "НК РФ": {...},
  "КоАП РФ": {...},
  "СК РФ": {...},
  "ЗК РФ": {...}
}
```
Ровно те 6 кодексов, что покрывают 9 практик (ФЗ об ООО/АО и 152-ФЗ — отдельные
списки номеров статей, без полных названий, т.к. это не кодексы с сквозной
нумерацией разделов). Индекс хранит только номер+название статьи, не полный
текст — достаточно для проверки существования, не раздувает репозиторий.

### Инструмент
`lookup_article(law: str, number: str) -> {"exists": bool, "title": str | None}`.

### Транспорт
Локальный MCP-сервер (stdio), поднимается как обычный дочерний процесс из
`citation_verifier.py` при первом вызове и переиспользуется (не поднимается
заново на каждый запрос).

## Изменения в существующих файлах
- `app.py`: три роута (`/api/chat`, `/api/documents/generate`,
  `/api/contracts/analyze`) вызывают `orchestrator.handle_*` вместо прямых
  `classify_practice`/`call_llm`.
- `llm.py`: `classify_practice` и `call_llm` остаются как есть — становятся
  внутренними примитивами, которые использует `orchestrator.py`.
- Новые файлы: `orchestrator.py`, `citation_verifier.py`,
  `mcp_servers/law_lookup/server.py`, `mcp_servers/law_lookup/law_index.json`,
  `templates/NDA.txt`, `templates/Договор.txt`, `templates/Претензия.txt`,
  `templates/Исковое заявление.txt`.

## Обработка ошибок (сквозной принцип, не меняется)
Каждый агент либо возвращает результат, либо деградирует безопасно — ни один
новый агент (классификатор шаблонов, верификатор цитат) не может превратить
рабочий ответ в отказ пользователю. Единственный источник настоящих ошибок
пользователю — сам LLM-вызов у SpecialistAgent (как и сейчас, `502` с понятным
текстом).

## Тестирование / проверка корректности
1. **PracticeClassifierAgent** — уже проверялся вручную (07-13): вопрос про
   увольнение → `labor`, вопрос про доли ООО → `corporate`. Без изменений.
2. **TemplateSelectorAgent** — юнит-тест: для каждого из 4 `doc_type` файл
   шаблона существует и непустой; для неизвестного `doc_type` возвращает `None`
   без исключения.
3. **CitationVerifierAgent** — юнит-тесты на regex-экстракцию (набор строк с
   ожидаемыми найденными цитатами) + юнит-тест на сам lookup: заведомо
   существующая статья (`ст. 15 ГК РФ`) → `exists: true`; заведомо
   несуществующая (`ст. 99999 ГК РФ`) → `exists: false` → предупреждение в
   выходном тексте.
4. **JuristOrchestrator** — интеграционный тест на каждый из 3 методов:
   мокнутый `call_llm`, возвращающий текст с одной верной и одной
   выдуманной статьёй → на выходе `handle_*` должно быть ровно одно
   предупреждение, относящееся к выдуманной статье.
5. **Ручная проверка (`/verify`)** — как и раньше: реальный вопрос по каждой
   из затронутых практик, реальная генерация документа с шаблоном,
   реальный анализ договора — но теперь дополнительно: намеренно
   спровоцировать модель сослаться на статью с явно неверным номером и
   убедиться, что предупреждение появляется в ответе.

## Границы (сознательно не делаем сейчас)
- Полнотекстовая база законов (только индекс номер+название — этого
  достаточно для отлова галлюцинаций, не для проверки актуальности нормы).
- Live-поиск/RAG по реальным текстам законов (вариант B из брейнсторминга) —
  отклонён из-за дополнительной сетевой зависимости.
- Память/контекст между агентами сверх того, что уже передаётся explicit
  (practice_id, template) — оркестратор остаётся детерминированной
  последовательностью вызовов, не «переговаривающимися» агентами.
- Мониторинг законов/новостей, поиск дел в VDR, GDPR/CCPA — как и в MVP-спеке,
  вне скоупа.
